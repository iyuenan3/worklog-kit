# 全仓深度 review 方法论（占位，不产品化）

> **定位**：本文是开发工序笔记，不是产品功能。PRD §14 已明确「全仓深度 review 方法论产品化」为不做项：它是开发期质量工序、token 成本高（单轮数十万至数百万 token），用户 vault 的日常质量由 worklog-lint（正误 + 体检）与 worklog-maintain 覆盖，不需要这套重炮。

## 方法骨架（本仓开发期实践了六轮）

1. **多视角并行审查**：每轮换一组全新维度（执行歧义 / 恶意输入 / 时间并发 / 长期规模 / 越轨用户 / 忠实度实跑 / 合规 / 威胁建模 / 双语一致 / 通用性红线……），同一批改动被 6 至 8 个互不知情的视角独立过一遍。
2. **逐条对抗验证**：每条发现交给多个不同视角的验证者尝试驳倒（事实核查 / 既定决策比对 / 实际影响评估），多数存活才算确认；驳倒项记录理由防止复发。
3. **确认全处置 + 单 commit**：确认项一次修复轮处置完，CHANGELOG 记「第 N 轮 review 修复」块（确认数 / 驳倒数 / 分级摘要）。
4. **修后复核**：修复本身再过一轮独立验证（逐条修复核对 + 横切一致性检查），抓「修 A 漏同步 B」类残留。
5. **实跑优先**：涉及执行正确性的轮次用 fixture vault 实跑（E2E + ground truth 对照），不靠读代码想象。

## 为什么不产品化

- 成本与收益错配：用户 vault 是个人数据仓，日常腐坏由体检五项覆盖；深度 review 针对的是「作为产品分发的指令面」，只有 kit 开发者需要。
- 已有轻量替代：worklog-lint 正误节 + 健康节 + worklog-maintain 修复闭环。
- 历史记录：六轮实践的完整结果见 `AIREADME/CHANGELOG.md` 各「review 修复」块。
