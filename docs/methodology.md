# 项目生命周期工作流

> 一个人同时跑很多项目时，怎么用一个「父项目」当大脑，把每个项目从想法到退役都管住。
> 这是把 worklog-kit 四个 skill（[`aireadme`](../.claude/skills/aireadme/) / [`stash`](../.claude/skills/stash/) / [`pitfalls`](../.claude/skills/pitfalls/) / [`worklog-ingest`](../.claude/skills/worklog-ingest/)）串成一条工作流的方法论。它们不是孤立工具，是这条产线上的工位。

---

## 思想：为什么需要一个「父项目」

一个人同时推进十几个项目时，真正的瓶颈不是写代码，是**上下文**：每个项目的需求为什么这么定、上周那个决策的理由、三个月前在另一个项目踩过的坑，散落在脑子、笔记本、各个仓库的 commit 里，谁也串不起来。换个项目就断片，换台机器就丢失。

第一性原理上，最干净、最省力的解法是：**让一个「什么都知道」的地方当大脑。** 这个大脑本身也是一个项目，但它是所有项目的**父项目**，其他项目都是它的子项目。子项目各自闷头开发，父项目负责：把想清楚的想法孵化成项目、给新项目立项、每天收集所有子项目的动态、沉淀跨项目的知识。

这个父项目这里叫 **worklog**（一个个人知识库 + 工作日记系统）。它不是一个旁边的笔记工具，是整套方法的中枢。

---

## 父项目 worklog 的三重身份

worklog 同时是三样东西，这三重身份正好覆盖一个项目「从生到死」需要的全部支撑：

1. **孵化器**：新项目从这里的对话里长出来。在父项目里把需求、定位、关键决策聊透，成熟了才分出去立项。
2. **知识库（Wiki）**：所有子项目的结构化知识（架构、决策、关系、部署）沉淀在这，跨项目可查、可互相引用。
3. **日记**：每天把所有活跃子项目的动态（commit、决策、踩坑、对外动作）收集成可复盘的日记。

子项目各有自己的真相源（见下「三只手」），但它们的**诞生、每日活动、知识沉淀**全部回流到父项目 worklog。父项目是起点，也是每天的汇流口。

## 子项目的生命周期

一个子项目不是一条直线走到死，是个**状态机**。「暂停」才是常态，「归档」是少数。

```
⓪ 捕获 ─深思─▶ ① 孵化 ──▶ ② 立项 ──▶ ③ 活跃 ⇄ 暂停 ──▶ ④ 归档（少数）
  拒绝多数        可流产     骨架+aireadme   每天 ingest 汇流   触发 = 交付完 / 回看判不再值得续
```

### ⓪ 捕获（漏斗的入口，在父项目之外）

想法是突然冒出来的。第一要务是**零摩擦地接住、不丢火花**：手边任何能记的地方都行（便签、手机备忘、聊天给自己发一条）。这一步刻意不进父项目：父项目不是 inbox，是漏斗的**出口**。有空了再对着这个想法深入想一遍，想清楚「值得认真做」，才带进父项目开始孵化。

**漏斗的主产物其实是「拒绝」。** 捕获的想法绝大多数会在这道「深思」闸门被毙掉，只有少数提纯出来进孵化。这不是浪费，恰恰是漏斗的本职：它存在的意义就是筛，让父项目只接「值得认真做」的那几个，而不被一堆半成品想法淹没。

### ① 孵化（worklog 对话）

在父项目里和 AI 把需求、定位、关键决策聊透。这一步的产出是「孵化种子」：一份想清楚了的需求 + 决策 + 项目身份（名字、类型、边界）。**做工具给别人用时，这一步要追问到底**（对方现在用什么、真实痛点是什么、哪些是「不会用」而非「没有」），否则立出来的是个自我想象的产品。

孵化也可能流产：聊着聊着发现不值得做、或问题根本不成立，就停在这一步、不立项。把一个想不清的想法挡在立项之前，比立项后再推倒便宜得多。

### ② 立项（aireadme）

需求稳定，就把项目正式立出去：**先铺最小骨架**（README + 一份需求文档 + 项目说明 + `git init`），不预写部署配置那些现场才定的东西。然后用 [`aireadme`](../.claude/skills/aireadme/) skill 给新项目建 `AIREADME/`：一套 AI 原生的项目文档体系，作为这个子项目自己的**真相源**（架构、决策、契约、部署各归其位，AI 和人都照它走、不跑偏）。

立项完，把新项目登记进父项目的扫描范围，从此每晚的收集会带上它。

### ③ 活跃 ⇄ 暂停（开发主循环）

开发期间：写代码、用 [`stash`](../.claude/skills/stash/) 把项目级的决策和踩坑存进项目记忆、架构变了就用 aireadme 更新真相源。

关键是**「暂停」是一等状态**，不是失败。自己的项目忙不过来时，不强行推进、也不杀掉，就**暂停**，有空再续。它仍是活的，只是这段时间没动静。这反直觉但很实用：逼自己「要么做完要么放弃」，反而会因为压力做出更差的取舍。

但「暂停能当一等状态」是有前提的，而这个前提恰恰是父项目：**暂停之所以安全，是因为父项目把上下文冻住了、能随时取回。** 决策的理由在 aireadme，那段时间的来龙去脉在日记，踩过的坑在项目记忆里。半年后回来，读一遍就接上，不必从零重建。没有这套基建，「暂停」就等于「腐烂」：回来时早忘了当初为什么那么定、做到了哪一步，续不上，只能重做。

所以「暂停是一等状态」不是一种豁达的态度，是父项目基建的**直接红利**：它把「保留以后再做这个选择权」的持有成本压到几乎为零。这正是实物期权的逻辑：持有成本既然几乎为零，留着选择权的下行就很有限，却保住了未来续做的全部上行。

### 横切 · 每晚收尾（worklog-ingest，贯穿 ③ 活跃的日常动作，不是一个独立状态）

每天收尾时用 [`worklog-ingest`](../.claude/skills/worklog-ingest/) skill，跨**所有**活跃子项目把这一天聚合成日记：扫各项目 commit、收集决策与踩坑、更新 Wiki、提交（有进展的晚上才有日记可聚，没动静的夜晚自然跳过）。这是 N 条并行的子项目生命周期每晚的**汇流口**：不管白天在几个项目间跳，晚上都回到父项目沉淀成一条可复盘的记录。

收尾时还顺手做一件「捕获」：把当天各项目新踩的、闻着像通用的坑标记成晋级候选（只打标、不下判断）。至于这些候选到底升不升进公共库，是另一道**定期**（而非每晚）的工序，理由见下「集体记忆」。

### ④ 归档

终点有两种，性质不同。**客户项目交付即归档**：这是成功的终点，活跃着直接交付收尾、不经暂停。**自己的项目判定不再值得做，才主动归档**：这种才稀少。两种都走同一套归档流程（Wiki 标记 + 移出活跃区）。判死型归档可以从活跃来、也可以从暂停来：一个暂停的项目，某次回看判定不再值得续，就从暂停转归档（触发是「回看判不值得」这个价值判断，不是「暂停得久」，暂停本身可以无限期）。所以除了客户交付这条成功终点，判死型归档始终是少数：「忙不过来」只会进暂停，不会进归档。

## 三只手：三个尺度的真相源

父项目伸向子项目的三只手，各自维护一个**不同尺度**的真相源。这是整套方法在工程上的核心：**文档与记忆不是一锅烩，按尺度分层治理。**

| skill | 尺度 | 真相源在哪 | 管什么 |
|---|---|---|---|
| [`aireadme`](../.claude/skills/aireadme/) | **项目尺度** | 每个子项目仓自己的 `AIREADME/` | 这个项目的架构、决策、契约、部署 |
| [`worklog-ingest`](../.claude/skills/worklog-ingest/) | **活动尺度** | 父项目 worklog 的日记 + Wiki | 跨所有项目的每日动态、长期知识 |
| [`stash`](../.claude/skills/stash/) | **会话尺度** | Claude Code 的项目记忆目录 | 一段对话里值得跨会话留存的决策、偏好、坑 |

三者不重叠：项目内的架构契约归 aireadme、跨项目的活动归 worklog、会话里的临时学习（含踩坑）归 stash。一条信息在任一时刻只落一个尺度（跨尺度的不是同时存两份，是下面说的「晋级」流向）。

更准地说，它们落在两条正交轴上：**跨项目 vs 项目本地**（只有 worklog 跨项目，aireadme 与 stash 都在项目本地）+ **策展 vs 原始**（aireadme 是策展后的项目真相，stash 是会话里的原始捕获）。所以晋级是有方向的：会话里反复出现、稳定下来的决策，从 stash 毕业进 aireadme 的决策记录；跨项目反复用到的，再回流 worklog Wiki。坑也走一条同样形状的阶梯（会话原始捕获 → 策展共享 → 全局蒸馏），只是它跨项目、专管踩坑，那层跨项目的共享坑库就是 [`pitfalls`](../.claude/skills/pitfalls/)，下一节细说。

---

## 集体记忆：一个项目踩坑，整个舰队免疫

父项目结构带来的最大杠杆，不是孵化能力，是**集体记忆**。

默认情况下，记忆是**项目隔离**的：A 项目里学到的工程教训，B 项目的 session 看不见。于是同一个 macOS locale 坑、同一个 git 坑、同一个部署坑，被不同项目轮流踩一遍。单人跑十几个项目，这是巨大的隐性浪费。

解法是给坑分三层、设一条**提纯阶梯**：

| 层 | 是什么 | 存哪 | 谁投递 |
|---|---|---|---|
| **单项目坑** | 只对单个项目成立 | 该项目的会话尺度记忆（[`stash`](../.claude/skills/stash/) 写，aireadme 管架构契约、不管踩坑） | 项目内 session 自动加载 |
| **通用工程坑** | 任何项目都可能踩 | [`pitfalls`](../.claude/skills/pitfalls/) 的库 | 干高危活前主动查 |
| **全局铁律** | 最高频最致命的少数 | 全局指令（每个 session 自动加载） | 永远在场 |

这三层怎么流转、由谁晋级，一张图看全：

```
   项目A        项目B        项目C        记忆项目隔离：A 看不见 B 踩过什么坑
     ╲           │           ╱           各项目把坑 stash 进自己的项目记忆
      ▼          ▼          ▼
   ▶ 父项目 worklog ◀     唯一同时握着所有项目坑的人，
     │                    才看得见「跨项目复现」、才做得了晋级
     │  定期晋级扫描 = 两问闸门 + 泛化 + 脱敏（私有 → 公开）
     ▼
  ┌─ 三层提纯阶梯（越往上越贵、越克制）──────────────────
  │   第 3 层  全局铁律     每 session 自动加载，永远在场（被动免疫）
  │   第 2 层  通用工程坑   pitfalls 共享库，高危活前主动查（接种）
  │   第 1 层  单项目坑     项目记忆（stash），项目内 session 自动加载
  └─ ↑ 通用坑升级 / ↓ 失准的铁律降级 ────────────────
     │
     ▼  投递回各项目
  舰队免疫：任何项目动同类活前一查就避开，不必每个项目各踩一遍
  ⚠ 升错的坑也会同样传遍全舰队（自身免疫病）→ 共享坑须可证伪 + 定期复检
```

图里那条阶梯，关键动作只有一个：踩到坑先判「单项目还是通用」，通用的升进 [`pitfalls`](../.claude/skills/pitfalls/)，往后任何项目动同类活之前查一眼，就**对这个坑免疫了**。

举个具体的：一个 macOS grep 默认 locale 漏中文匹配的坑，最早只记在某个项目的记忆里；抽象进 [`pitfalls`](../.claude/skills/pitfalls/) 后，后来的项目在动 grep 之前一查就避开了，没有第二个项目再踩同一个。这就是「舰队免疫」：把单个项目用血换来的教训，变成所有项目的共享抗体。父项目 = 集体记忆的载体，这条阶梯 = 抗体的投递机制。

**知识也有生命周期，和项目对称。** 那条提纯阶梯不只是单向往上走：某条升上去的「铁律」，若后来发现其实只在特定场景成立，就该**降级**回去，否则它会把局部经验冒充成普适规则、误导所有项目。一条教训从会话里的原始捕获，到策展共享，到全局蒸馏，本身就是一段生老病死，和上半篇的项目状态机正好对称。

**晋级要有闸门，否则共享库会被自己撑垮。** 不是每个坑都配往上升。判断三问：**它跨项目会复现吗？它非显而易见吗？它的根因已复现、已验证吗？** 三个都「是」才升进 [`pitfalls`](../.claude/skills/pitfalls/)。前两问管「值不值得共享」，第三问管「这条本身对不对」：共享库里一条没验证的错坑会和对的一样全舰队传播（见下「自身免疫」），错的比没有更糟，所以根因还在「疑似」、没复现过的坑，留本项目、先别外发。把只对单项目成立的、或凭常识就能避开的也一股脑塞进去，共享库就从「抗体库」退化成「噪音堆」：条目一多，查的人翻不动、也不再信任，于是干脆不查，整个机制空转。**共享记忆的价值在精、不在多**，克制地拒绝晋级，和努力地收集同样重要。还有一道跨边界的手续：[`pitfalls`](../.claude/skills/pitfalls/) 是公开的，升进去之前得**泛化加脱敏**，剥掉项目 / 客户的私有信息（真正通用的坑泛化后本就不该含这些，那部分恰恰是「项目本地」、压根不该升）。

**晋级由谁做？只能是父项目。** 有的坑一照面就明显通用（与具体项目无关、纯是工具或系统的脾气），照闸门当场就升。但更多坑当时看着像「就这个项目的事」，得靠**跨项目复现**才暴露通用性：同一个坑在第二个项目又冒出来。而复现只有父项目看得见，因为**晋级之前 A 项目根本不知道 B 项目早踩过同一个坑**，记忆是项目隔离的。只有父项目，靠每晚 ingest 把各项目的坑都汇进日记，才攒下那份横跨所有项目的视图，才看得见两次撞在一起。

**什么时候做？定期，不是每晚。** 因为要等复现、等横向攒够才判得准，晋级是一道定期工序：每晚收尾只把候选标出来，真正的判断攒一批再做，而不是每踩一个坑就当场逼自己表态。这再次印证内核：跨项目的事，只有那个唯一俯瞰全局的父项目做得了。

**还有一条得说破：共享记忆有它独有的风险，错的坑也会全舰队传播。** 单项目记忆里写错一条，最多坑一个项目；但共享库里一条错断言，会和正确的那些一样被传遍整个舰队，让每个项目都去躲一个根本不存在的威胁，或对着一个伪问题套一个错的修法。这是集体免疫的反面，**自身免疫病**：系统调动全身去攻击一个其实无害的目标。所以集体记忆比单项目记忆**更**需要治理：每条坑都得可证伪（依赖它之前能照着复现一次）、定期复检（工具升级后旧坑可能早已不成立）、标清适用版本、并敢于剪枝。这不是空谈：这套库自己建起来时，一轮对抗式复核就抓出「BSD grep 不支持 `\b`」是一条过时的错断言（在当前版本的 grep 上根本不复现）。**一条没复检过的共享坑，比没有这条坑更危险。**

> ⚠️ 命门在「投递」、不在「存储」。把坑存进一个共享库，并不会让任何项目自动看见它：记忆是项目隔离的，库也不会在你正要踩坑的那一刻自己跳出来。真正让库活着的不是库本身，是**投递机制**：一个可被主动查的 [`pitfalls`](../.claude/skills/pitfalls/) 库，加一行「干高危活前先查」的全局纪律（它本身就是一条全局铁律，只是管的是投递、不是躲坑内容；每个 session 都加载、永远在场）。少了这条纪律，再全的库也只是一座墓地，躺着一堆没人在对的时刻翻开的教训。

## 小结

这套工作流的内核是一句话：**用一个父项目当大脑，把一群项目的生老病死管起来。**

- 父项目（worklog）是中枢：孵化器 + Wiki + 日记。
- 子项目走一条状态机：捕获 → 孵化 → 立项 → 活跃⇄暂停 → 归档（暂停是常态，归档是少数）。
- 三只手按尺度分层维护真相源：aireadme（项目）/ worklog-ingest（活动）/ stash（会话）。
- 集体记忆让一个项目的教训变成整个舰队的免疫力，但错的教训也会同样传播，所以靠晋级闸门和定期复检来治理。

四个 skill 都随 worklog-kit 开源（`.claude/skills/` 下；[`aireadme`](../.claude/skills/aireadme/) / [`stash`](../.claude/skills/stash/) / [`pitfalls`](../.claude/skills/pitfalls/) 可单独取用装到 `~/.claude/skills/`，worklog-ingest 为 config 驱动的通用实现）。

**最小上手**：worklog-kit 本身就是这条工作流的开箱实现：用模板建你的父项目 vault、跑 `/worklog-init`，三件套与全局纪律自动装好，每晚一句话汇流。只想要局部的，也可以只装 aireadme + stash + pitfalls，手动记日记、手动标晋级候选，跑一个精简版。合起来，这是一个人把很多项目跑得不丢上下文的一套方法。
